# Xilinx/dma_ip_drivers github의 read/wrtie 분석

## 서론
- PCIe(Peripheral Component Interconnect Express)는 기존 병렬 버스 방식이었던 PCI와 PCI-X의 한계를 극복하고 대역폭과 확장성을 높이기 위해 개발된 고속 인터페이스 기술.
- 오늘날 거의 모든 PC, 서버, 산업용 컴퓨터에서 주변기기를 연결하는 핵심 표준으로 사용.

## PCIe 핵심 개념
- 직렬 및 점대점 연결 (Serial & Point-to-Point): 여러 기기가 하나의 버스를 공유하던 기존 PCI 방식과 달리, 두 기기가 1:1로 직접 연결되는 점대점(Point-to-Point) 직렬 통신 방식을 사용.
- 링크(Link)와 레인(Lane): 두 기기간의 물리적 연결 통로를 **링크**라고 하며, 링크는 하나 이상의 **레인**으로 구성.
- 각 레인은 송신(Transmit)과 수신(Recevice)를 위한 차동 신호(Differential Signal) 쌍으로 구성되며 필요에 따라 레인을 x1, x2, x4, x8, x16, x32개로 묶어 대역폭과 성능을 유연하게 확장할 수 있음.
- 듀얼 심플렉스 (Dual-Simplex): 데이터의 송신과 수신 경로가 독립적으로 존재하여 동시에 데이터를 주고받을 수 있는 듀얼 심플렉스 연결을 제공.
- 패킷 기반 프로토콜 (Packet-Based Protocol): 전통적인 버스에서 사용되던 다양한 제어 신호 핀을 없애는 대신, 모든 정보와 데이터를 패킷(Packet) 구조로 만들어 전송.
- 공통 클럭 제거(No Common Clock): 별도의 외부 동기화 클럭 신호를 보내지 않음. 대신 송신기가 데이터 스트림에 클럭을 내장(Embedding)하여 보내면, 수신기 내부의 PLL(Phase-Locked Loop)회로가 데이터에서 클럭을 복원하여 데이터를 수신.
- 소프트웨어 하위 호환성(Software Backward Compatibility): 하드웨어 통신 방식은 직렬로 완전히 바뀌었지만, 메모리 공간, I/O 공간, 설정 공간(Configuration Space)등 기존 PCI의 소프트웨어 프로그래밍 모델을 그대로 유지.

## PCIe 시스템 연결 구조 (Topology)
- PCIe는 점대점 연결만 가능하므로, 여러 기기를 연결하기 위해 스위치 등을 활용한 트리(Tree) 형태의 토폴로지(Topology)를 구성. 
- 루트 컴플렉스 (Root Complex, RC): CPU 및 시스템 메모리와 PCIe 디바이스(주변기기)들을 연결해 주는 최상위 계층의 인터페이스
- 스위치 (Switch): 하나의 포트에 여러 디바이스를 연결할 수 있게 팬아웃(Fanout) 기능을 제공하는 라우터 역할을 함. 패킷의 주소나 라우팅 정보를 확인하여 올바른 경로로 패킷을 전달함.
- 브리지 (Bridge): PCIe 네트워크에 구형 PCI나 PCI-X 등 다른 규격의 버스를 연결할 수 있게 해주는 변환 인터페이스.
- 엔드포인트 (Endpoint): PCIe 트리 구조의 맨 끝부분에 연결되는 최종 주변기기를 말함. 스스로 트랜잭션을 시작하거나 응답하는 역할을 함.

## PCIe 통신 계층 구조 (Layered Architecture)
- PCIe는 다비아스 간의 통신을 효율적으로 처리하기 위해 역할별로 3개의 논리적 계층(Layer)으로 나뉘어 작동.
- 트랜잭션 계층 (Transaction Layer):
    - 최상위 계층으로 디바이스의 코어(소프트웨어 계층)와 직접 통신.
    - 데이터 읽기/쓰기 등의 명령을 담은 TLP(Transaction Layer Packet)를 생성하고 목적지에서 해독하는 역할을 함.
    - 데이터의 흐름을 관리하는 가상 채널(Virtual Channel), 트래픽의 우선순위를 정하는 QoS, 트랜잭션 순서 유지(Odering) 규칙을 담당.
- 데이터 링크 계층(Data Link Layer):
    - TLP 패킷이 링크를 통해 손상 없이 안전하게 전송되도록 보장하는 역할을 함.
    - 이를 위해 TLP 앞뒤에 시퀀스 번호와 LCRC(Link CRC)를 붙여 전송하고, 수신 측에서 에러를 검사.
    - 오류가 없으면, Ack, 오류가 있으면 Nak을 보내 재전송을 요청하는 Ack/Nak 프로토콜을 수행하며, 수신 버퍼의 여유 공간을 관리하는 흐름 제어(Flow Control)도 담당.
- 물리 계층 (Physical Layer)
    - 가장 하위 게층으로 패킷을 실제 전기적 신호로 바꾸어 케이블이나 보드 패턴을 통해 전송하는 역할을 함.
    - 데이터 바이트를 여러 레인으로 분산시키는 스트라이핑, EMI를 줄이기 위한 스크램블링, 클럭 복원을 위한 8b/10b 또는 128b/130b 인코딩 작업을 수행한 뒤 병렬 데이터를 직렬 비트 스트림으로 변호나하여 전송.
    - 링크의 속도 및 폭을 협상하고 초기화하는 트레이닝(LTSSM) 과정도 이 게층에서 수행.

## Configuration Overflow (설정 공간 및 토폴로지 발견)
- 디바이스가 시스템에 어떻게 연결되어 있는지 파악하고 초기화하는 과정에 대한 내용을 다룸.
- BDF(Bus, Device, Function): PCIe 시스템 내의 모든 기능(Function)은 버스(Bus), 디바이스(Device), 기능(Function) 번호의 조합인 BDF를 통해 논리적으로 고유하게 식별됨.
- 설정 공간(Configuration Space): 운영체제나 시스템 소프트웨어가 디바이스를 인식하고 설정하기 위해 사용하는 전용 주소 공간. 기존 PCI와 호환성을 유지하는 256바이트의 PCI 호환 공간과, PCIe에서 새롭게 추가된 기능을 지원하기 위해 확장된 4KB 크기의 확장 설정 공간으로 구성.
- 설정 트랜잭션 생성 방식: 프로세서가 설정 공간에 접근하기 위해 두 가지 메커니즘을 사용.
    - 1. Legacy PCI 메커니즘: 제한된 IO 공간을 아끼기 위해 IO 포트 두개(CF8h 주소 포트, CFCh 데이터 포트)를 이용해 간접적으로 접근하는 방식.
    - 2. Enhanced Configuration Access 메커니즘: 멀티 코어/멀티 스레디 환경에서 발생할 수 있는 충돌을 막기 위해 4KB 설정 공간 전체를 시스템 메모리 주소(256MB 크기)에 직접 매핑하여 단일 명령으로 안전하게 접근하는 방식.
- 열겨(Enumeration)와 토폴로지 발견:시스템 부팅 시 Root Complex는 깊이 우선 탐색(Depth-first search) 방식으로 하위 버스를 스캔하며 스위치(브리지)와 엔드포인트를 발견하고 순차적으로 버스 번호를 할당.
- Type 0 및 Type 1 설정 요청: 라우팅 시 목적지 버스가 브리지 바로 아래의 버스(Secondary Bus)이면 해당 디바이스가 처리하도록 Type 0 요청으로 변환되며, 목적지 버스가 그보다 더 하위에 있다면 브르지를 통과시키기 위해 Type 1 요청으로 전달. 헤더 타입 레지스터를 통해 해당 장치 엔드포인트인지 브리지인지 구분.

## Address Space
- 디바이스가 데이터를 주고받기 위해 주소를 어떻게 할당받고, 패킷(TLP)이 목적지를 어떻게 찾아가는지 다룸.
- 3가지 주소 공간: PCIe는 Configuration(설정), Memory(메모리), IO(입출력) 세 가지 주소 공간을 지원. 이 중 메모리 공간은 데이터를 읽을 때 상태가 변하지 않아 성능 향상을 위해 미리 읽기(Speculative fetch)가 가능한 Prefetchable(P-MMIO) 공간과, 부작용이 있어 불가능한 Non-Prefetchable(NP-MMIO) 공간으로 나뉨.
- BAR(Base Address Register): 디바이스가 자신이 필요로 하는 메모리나 IO 공간의 크기와 타입을 시스템에 요청하는 역할.
    - 소프트웨어는 먼저 BAR에 모든 비트를 1로 기록한 뒤 다시 읽어오는 방식을 통해 디바이스가 요청하는 주소 공간의 크기를 파악.
    - 크기를 파악한 후 시스템 소프트웨어는 가용한 시작 주소(Base Address)를 BAR에 기록하여 주소 할당을 완료. 엔드포인트는 최대 6개의 BAR를 브르지는 2개의 BAR를 가질 수 있으며, 모든 BAR는 순차적으로 평가되어야 함.




# Device Driver

## 디바이스 드라이버의 개념
- 운영체제(windows, Linux 등)는 하드웨어가 어떻게 생겼는지 다 알지 못함, 대신 드라이버가 하드웨어의 복잡한 레지스터 조작법을 숨기고, OS가 이해할 수 있는 표준 명령어(읽기, 쓰기 등)로 바꿔줌.
    - 상부 인터페이스: 유저가 read(), write() 같은 함수를 쓰면 드라이버가 이를 가로챔.
    - 하부 인터페이스: 드라이버가 실제 하드웨어 주소(0x1234...)에 전기적 신호를 보내 데이터를 주고 받음.
